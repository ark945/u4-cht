# Amiga 版 Ultima IV 音樂格式逆向手記(`mus*.bin`)

> 沒有公開規格、連 Amiga 社群都在「求檔」的情況下,從零逆出 Ultima IV(Origin, 1988)
> Amiga 版背景音樂的序列格式,把旋律抽出來。

這份手記記錄的是**音樂序列(songs)**的格式逆向;**音效(PCM samples)**的抽取另見
[各平台音效抽取手記](audio-extraction.md)。兩者是兩條不同的路:SFX 是現成取樣,直接轉 WAV;
songs 是給音源驅動跑的指令序列,要先逆出格式、再合成。

## 起點:沒有 spec

Origin 1988 的 Amiga U4 音樂用的是**自家私有格式**,沒有公開文件 —— 網路上甚至找得到
Amiga 論壇標題「**WANTED: Ultima IV Amiga Music**」的求檔串。所以這裡沒有 spec 可讀,
只能對著二進位逐 byte 拆。

檔案在 WHDLoad 包解開後的 `data/ultmapp/`:`musb.bin` / `musc.bin` / `musd.bin` /
`muso.bin` / `must.bin` —— 推測對應 **b**attle / **c**astle / **d**ungeon / **o**utdoor /
**t**owne 五個場景的曲子(各 1.5–3 KB,熵 ~5.3,屬序列資料而非取樣)。

## 拆出來的結構

### 1. 檔頭 = BE16 的 voice offset 表

開頭是一串 big-endian 16-bit 值。比對五首歌的開頭會發現固定模式:

```
must: 050b 0011 | 0070 0468 06d8 0701 | 0400 0000
musc: 050b 0011 | 0074 026c 04dc 0501 | 0400 0000
muso: 050b 0011 | 0058 0350 05c0 0601 | 0400 0000
```

`0070 0468 06d8 0701` 這幾個遞增、且都 < 檔案大小的值,就是**各聲部(voice)資料的起始
offset**。Amiga 的 Paula 音源是 **4 聲道**,對得上 4 個 offset。第一個 offset(`0x70`)正好
是檔頭結束、第一段事件資料開始的位置 —— 一次對上。

### 2. 事件流 = 命令 byte + 音符 byte

每個 voice 區段是一串事件。把某個聲部的前幾十 byte 攤開看:

```
... a4 32 00 80 06 92 32 86 33 84 35 33 32 a4 30 ...
```

規律是:**高位有設(`0x80`–`0xff`)的 byte 是命令**,**落在 `0x20`–`0x3f` 的 byte 是音符**。
把所有音符 byte 抽出來統計,值集中在 `0x30`(48)、`0x32`(50)、`0x33`(51)、`0x35`(53)、
`0x37`(55)…—— 換算成半音就是 **C / D / E♭ / F / G**,一個清楚的音階,不是雜訊。

命令 byte 的低位(尤其低 4 bit)推測是**音長 / 時值**,但確切編碼還沒完全定下來。

### 3. 用 piano-roll 驗收(沒喇叭時的眼睛)

光看 byte 還不能確定抽對了。把每個 voice 的音符序列畫成 **piano-roll**(x = 事件序、
y = 音高),一眼就看出來:

- **voice 0**:一條清楚的旋律線,音符成階梯狀起伏、有樂句 —— 這是**主旋律**。
- **voice 2**:較低、較持續的線 —— **低音 / 和聲**。
- voice 1 / 3:較短的片段(節奏 / 點綴)。

旋律的 contour(輪廓)是連貫的音樂線條,不是隨機跳動 —— 這就確認了「檔頭 offset 表 →
4 voice → 命令+音符事件」這套拆法是對的。

## 合成:從音符到聲音

抽出音符後,用最簡單的**方波合成**先把旋律放出來驗證:音符值當 MIDI 半音算頻率
(`f = 440 × 2^((note−69)/12)`),命令低位當音長,多聲部混音。五首都合成出**完整長度的曲子**
(183–320 秒),長度合理 —— 是真的整首歌,不是幾秒的碎片。

工具:
- `tools/amiga/audio/mus_explore.py` —— 拆檔頭 / voice、抽音符、畫 piano-roll。
- `tools/amiga/audio/mus_synth.py` —— 方波合成出 WAV(旋律證明版)。

## 解到哪、還沒解到哪(誠實邊界)

- ✅ **檔案結構**:檔頭 offset 表、4 voice 切分 —— 確定。
- ✅ **音符序列**:每個 voice 的音高序列抽得出來,piano-roll 確認是連貫旋律。
- 🟡 **時值 / tempo**:命令 byte 低位疑為音長,但確切編碼(以及全曲速度)還是近似值。
- 🟡 **音色**:目前用方波頂著;真實 Amiga 版用的是 Paula 取樣(8-bit PCM 樣本),要忠實
  重現得找到**音樂用的樣本庫**(`snds.bin` 是音效庫,音樂樣本是否同源待確認)+ 音高基準校正。
- ⛔ **保真重現**:要做到和原機一樣的聲音,需要 driver 級的 RE(命令集全解 + 樣本對映 +
  Paula 週期表)。目前是「**旋律抽取證明**」,不是保真版。

## 方法論(可移植到其他無 spec 的序列格式)

1. **多檔比對找檔頭**:同一格式的多個檔案,開頭固定的欄位就是 header 結構(這裡是 voice offset 表)。
2. **遞增且 < filesize 的值 = offset**:序列格式的檔頭常是指向各區段的 offset 表。
3. **值域分類 byte**:高位 set = 命令、落在連續小範圍 = 音符 —— 把音符抽出來看值是否成音階。
4. **piano-roll 驗收**:畫音高對事件序的圖,看 contour 是不是連貫旋律 —— 沒喇叭也能判對錯。
5. **先方波、後保真**:先用方波把旋律放出來證明音符對,音色 / tempo / 取樣留待 driver 級 RE。
